# Content / Runtime Split — Two-Project Architecture Design

## Overview

Split the Godot project into two cooperating Godot 4.6.2 projects:

- **`complete-app`** — thin runtime "player" shipped to web. Loads PCKs at runtime to manage WebAssembly memory pressure and enable hot-reload of content.
- **`content-app`** — content authoring + preview harness. Iterates on HUD, enemies, NPCs, items, dialogue, locations. Builds PCKs consumed by `complete-app`.

A single shared folder (`project/shared/`) outside both Godot projects holds addons, shared GDScript libraries, shaders, registries, and source content. Host projects reference it via OS-appropriate links (symlinks on Mac/Linux, directory junctions on Windows) so identical paths resolve under `res://` in both editors without admin privileges.

## Goals

1. **Memory** — Web build loads only the kernel + boot bundle initially; gameplay content loads on demand and unloads when not needed.
2. **Iteration speed** — Tweaking an enemy or HUD widget never requires rebuilding `complete-app`. Rebuild a single PCK, hot-reload, see change.
3. **Single source of truth** — Addons, shared scripts, shaders, and source content live in one place. No copy-paste drift.
4. **Cross-platform development** — Setup works identically on macOS and Windows without admin rights.
5. **Safe fallback** — `complete-app` can still launch and run the boot scene even if no PCKs are present (development-time graceful degradation).

## Non-Goals

- Multiplayer content syncing
- Per-user / per-region content variants
- Hot-reload of autoload singletons (Godot does not support this cleanly)
- Replacing the existing location PCK pipeline (it stays — this spec generalizes its pattern)
- CDN / asset versioning beyond a content hash in the manifest

### Why this layout

- **`shared/` outside host projects**: lets both projects see the same files via a *single* canonical path. No copy step, no drift.
- **Links not Godot-side path remapping**: Godot 4 expects addons at `res://addons/`. We can't redirect that without engine changes.
- **`content/` linked only into content-app**: avoids `.import` file ownership conflicts (only the authoring project re-imports raw assets).
- **`lib/resources/` shared by both**: Resource class scripts (e.g. `EnemyDefinition extends Resource`) MUST exist in the runtime so `.tres` files in PCKs deserialize correctly.
- **`scripts/` and `scenes/` real (not linked) in each host**: kernel scripts and the boot scene are unique per project — they don't belong in `shared/`.

## Cross-Platform Link Strategy

### The mechanism

| OS | Command | Admin needed? | Git behavior |
|---|---|---|---|
| macOS / Linux | `ln -s ../../shared/addons addons` | No | Tracked as symlink (with `core.symlinks=true`) |
| Windows | `mklink /J addons ..\..\shared\addons` | **No** (junctions are unprivileged) | Junctions appear as regular dirs to git — **must be gitignored** |

### Why junctions on Windows, not symlinks

- `mklink /D` (directory symlink) requires admin OR Developer Mode → blocks contributors
- `mklink /J` (junction) works for any user, same volume, no admin
- Junctions only work on the same drive — fine for a project tree
- Godot reads through junctions transparently

### Setup workflow

A new task wires this up:

```yaml
share:link:
  desc: Create cross-platform links from host projects to shared/
  cmds:
    - python3 scripts/setup_shared_links.py
```

`scripts/setup_shared_links.py`:
- Detects platform (`sys.platform`)
- For each `(host, link_name, target)` in a fixed table:
  - Removes any stale link/dir at `host/link_name`
  - Creates symlink (Mac/Linux) or junction (Windows)
- Idempotent — safe to re-run

Called from `task setup` so `task setup` is the only command needed for a fresh clone.

### Git behavior

`.gitignore` additions (in repo root):

```
# Cross-platform links to shared/ — recreated by `task share:link`
project/hosts/complete-app/addons/
project/hosts/complete-app/lib/
project/hosts/complete-app/shaders/

project/hosts/content-app/addons/
```

A pre-commit hook guards against accidentally committing files inside the linked paths (which would mean editing through the symlink instead of `shared/`):

```python

# Reject staged files matching project/hosts/*/{addons,lib,shaders,data,content}/...
```

## Component Details

### 1. Kernel (in `complete-app`)

Already-existing autoloads stay. One new autoload added:

| Autoload | Status | Purpose |
|---|---|---|
| `ECS`, `BeehaveGlobalMetrics`, `BeehaveGlobalDebugger`, `GUIDE` | existing | from addons |
| `GameActions`, `GameConfig`, `TimeService`, `ActivityLog` | existing | game-side autoloads |
| `LocationManager` | existing | location PCK loader |
| **`ContentManager`** | **NEW** | generic content-bundle PCK loader |

`ContentManager` is to bundles what `LocationManager` is to locations — a generalization of the same pattern. `LocationManager` continues to handle location-specific concerns (palette swap, era switching) and may delegate raw PCK loading to `ContentManager` in a future cleanup (not in this spec).

### 2. `ContentManager` API

**File:** `project/hosts/complete-app/scripts/content_manager.gd`

```gdscript
extends Node
# Autoloaded as ContentManager

signal bundle_loading(bundle_id: String)
signal bundle_loaded(bundle_id: String)
signal bundle_load_failed(bundle_id: String, reason: String)
signal bundle_unloaded(bundle_id: String)

# Read content_manifest.json on _ready and cache the bundle table.
func _ready() -> void: ...

# Eager bundles (declared in manifest with "policy": "eager") loaded automatically
# during boot, after WS connection is established.

# Lazy bundles loaded on demand:
func load_bundle(bundle_id: String) -> bool
func unload_bundle(bundle_id: String) -> bool
func is_loaded(bundle_id: String) -> bool
func loaded_bundles() -> Array[String]

# Resource lookup helpers — content code looks up by stable ID, not by res:// path:
func get_enemy_definition(id: String) -> EnemyDefinition
func get_npc_definition(id: String) -> NpcDefinition
func get_hud_widget_scene(id: String) -> PackedScene
# (one helper per content kind defined in shared/lib/enums/content_bundle_kind.gd)

# Returns the manifest entry for debugging:
func describe(bundle_id: String) -> Dictionary
```

Internals:
- `ProjectSettings.load_resource_pack(pck_path, replace_files=false)`
- Tracks loaded bundles + their declared resource roots
- Honors a `dependencies` array in the manifest — a bundle's deps are loaded first
- Refuses to unload a bundle that another loaded bundle depends on
- On web: PCKs fetched via HTTP; relative paths in manifest are relative to a configurable `pck_base_url`

### 3. Content manifest

**File:** `project/shared/data/content_manifest.json`
**Generated by:** `task content:manifest` (reads `shared/content/**` + per-bundle `bundle.json` files)
**Consumed by:** `ContentManager` at runtime

```json
{
  "schema_version": 1,
  "bundles": {
    "hud-core": {
      "kind": "hud",
      "policy": "eager",
      "pck": "hud-core@a3f1.pck",
      "deps": [],
      "provides": {
        "scenes": ["hud_main", "hud_status_bar"],
        "widgets": ["activity_log_panel", "minimap"]
      }
    },
    "hud-battle": {
      "kind": "hud",
      "policy": "lazy",
      "pck": "hud-battle@b291.pck",
      "deps": ["hud-core"],
      "provides": { "scenes": ["combat_overlay"] }
    },
    "enemies-forest": {
      "kind": "enemies",
      "policy": "lazy",
      "pck": "enemies-forest@c5e0.pck",
      "deps": [],
      "provides": { "definitions": ["goblin", "slime", "wolf"] }
    },
    "whispering-woods": {
      "kind": "location",
      "policy": "lazy",
      "pck": "whispering-woods@d811.pck",
      "deps": ["enemies-forest"]
    }
  }
}
```

The `@hash` suffix is a content hash for cache busting in web builds. The manifest is the only file that knows the current hashed filename.

### 4. Bundle packager (in `content-app`)

**File:** `project/hosts/content-app/scripts/bundle_packager.gd`

Run via `task content:build -- {bundle-id}`. Headless GDScript script:

1. Reads `shared/content/{kind}/{bundle-id}/bundle.json` (declares included paths, provided IDs, deps)
2. Validates referenced resources (`.tres`/`.tscn`) parse cleanly
3. Computes content hash over included files
4. Calls `PCKPacker.pack()` with `shared/content/{kind}/{bundle-id}/**` → `build/_artifacts/pck/{bundle-id}@{hash}.pck`
5. Updates `shared/data/content_manifest.json` with the new entry (hashed filename + hash + provides list)

Per-bundle declaration: `shared/content/{kind}/{bundle-id}/bundle.json`

```json
{
  "id": "enemies-forest",
  "kind": "enemies",
  "policy": "lazy",
  "deps": [],
  "include": ["**/*.tres", "**/*.tscn", "**/*.png"],
  "provides": {
    "definitions": ["goblin", "slime", "wolf"]
  }
}
```

### 5. Preview harness (in `content-app`)

`preview/preview_main.tscn` is a developer menu: pick a content kind → pick an instance → opens an isolated preview scene with `mock_kernel.gd` substituting for autoloads.

`preview/` is **excluded** from all PCKs by the bundle `include` patterns (preview lives outside `shared/content/`), so it never ships.

This is what makes content-app actually useful for tweaking — open `goblin.tres`, hit Play, see the goblin in fake combat with mocked WS messages.

### 6. Boot flow (`complete-app`)

`scripts/boot.gd` orchestrates startup:

```
1. Show splash (boot.tscn)
2. ContentManager._ready loads content_manifest.json (HTTP fetch on web)
3. WS client connects to game server
4. ContentManager loads all "policy": "eager" bundles (typically just hud-core)
5. Wait for first server state snapshot
6. Server snapshot indicates current location → LocationManager.load_location()
7. Server snapshot indicates current screen (overworld / combat) → ContentManager.load_bundle("hud-battle") if entering combat, etc.
8. Hide splash, show gameplay UI
```

If `content_manifest.json` is missing or no PCKs are found:
- Boot scene shows a "no content available — running in dev fallback" banner
- Server still drives state; UI shows minimal text-only debug view
- This is the **safe fallback** — `complete-app` never hard-fails on missing content

### 7. Hot-reload workflow (development)

In dev mode (detected via `OS.has_feature("editor")` or env flag):

1. `ContentManager` watches `build/_artifacts/pck/` (filesystem watcher in editor; HTTP polling in served web)
2. On detected change to a loaded bundle's PCK:
   - Emit `bundle_will_reload(id)` so consumers can clean up references
   - Unload, re-load
   - Emit `bundle_loaded(id)`
3. UI scenes that hold references to bundle resources subscribe to `bundle_will_reload` and refresh

Hot-reload is opt-in per consumer. Scenes that don't subscribe simply keep their pre-reload references (still valid — Godot's resource cache holds them). New scene instantiations get the new content.

## Build Pipeline

### New / changed Taskfile commands

```yaml
share:link:
  desc: Create symlinks/junctions from host projects to shared/
  cmds: [python3 scripts/setup_shared_links.py]

content:manifest:
  desc: Regenerate content_manifest.json from all bundle.json files
  cmds: [python3 scripts/content_manifest.py]

content:build:
  desc: Build a single content bundle into a PCK (use -- {bundle-id})
  cmds:
    - python3 scripts/run_godot_checked.py
        {{.GODOT_PATH}} --headless
        --path project/hosts/content-app
        --script scripts/bundle_packager.gd
        -- --bundle={{.CLI_ARGS}}
    - task: content:manifest

content:build:all:
  desc: Build every content bundle declared under shared/content/**/bundle.json
  cmds: [python3 scripts/build_all_bundles.py]

# Existing commands updated to include shared link setup:
setup:
  cmds:
    - # ... existing setup steps ...
    - task: share:link

# Existing pck:build (locations) is kept as-is — it's a specialization
# that this generic pipeline can absorb later, but not in this spec.
```

### Output convention

```
build/_artifacts/
├── pck/
│   ├── hud-core@a3f1.pck
│   ├── hud-battle@b291.pck
│   ├── enemies-forest@c5e0.pck
│   ├── whispering-woods@d811.pck       # location PCKs land here too
│   └── manifest.json                    # copy of shared/data/content_manifest.json at build time
└── {version}/web/                       # Godot web export (kernel only)
    ├── index.html  index.wasm  ...
    └── pck/  → symlink or copy of build/_artifacts/pck for serving
```

Web serve: static server serves both the WASM kernel and the `pck/` directory. `ContentManager` fetches `pck/manifest.json` then individual PCKs as bundles are requested.

## Migration Plan (for the implementation plan, not this spec)

This spec defines the target state. The implementation plan will sequence:

1. Create `project/shared/` and move addons into it
2. Create the link script + run `task share:link`
3. Move shared scripts (Resource defs, contracts) to `shared/lib/`
4. Stand up `content-app` project.godot with autoloads matching `complete-app`
5. Build the preview harness scaffolding
6. Move existing scenes → `shared/content/` and split into bundles
7. Add `ContentManager` autoload + manifest pipeline
8. Wire `boot.gd` to load eager bundles + show fallback when manifest missing
9. Migrate location PCK pipeline to live alongside (no behavior change required)
10. Update web build to serve the `pck/` directory

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Resource class scripts diverge between projects | Both projects link to identical `shared/lib/` — single source. Pre-commit guard prevents writes to linked paths from host projects. |
| `.import` conflicts on shared content | Only `content-app` links `shared/content/`. `complete-app` sees content only via PCK at runtime. |
| Junction limitations on Windows (same drive only) | Acceptable — repo lives on a single drive in normal dev setups. Doc this in README. |
| Web HTTP fetch of many PCKs slows boot | Eager bundles are kept minimal (just `hud-core`). Lazy bundles only fetched on demand. |
| Hot-reload of resource that's currently in scene tree | Opt-in `bundle_will_reload` signal lets consumers refresh; non-subscribers keep stable references via Godot's resource cache. |
| PCK content references a class only present in another (unloaded) PCK | Manifest `deps` array enforces load order; `ContentManager` refuses unload while dependents loaded. |
| Godot autoload changes between projects desync | Both `project.godot` files are committed and reviewed; CI check (future) compares autoload lists for the kernel set. |
